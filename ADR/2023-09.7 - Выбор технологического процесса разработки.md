**Дата:** 24 декабря 2025 г.  
**Статус:** Принято

# Контекст

Проект развивается итеративно: сначала ядро (котировки, ордера, портфель), затем аналитика, стратегии, уведомления, интеграции.
Система 24/7 и связана с финансовыми операциями, поэтому важны:

- предсказуемость поставок и контроль качества;
- быстрые реакции на инциденты/срочные багфиксы;
- прозрачность работ между ролями (backend, frontend, QA, архитектор).

# Рассмотренные варианты

- **Waterfall** — слишком негибко для эволюции требований и частых релизов.
- **Scrum** — прогнозируемые спринты и понятные ритуалы, но хуже для незапланированных срочных задач.
- **Kanban** — хорошо для потока и инцидентов, но может теряться планирование крупных фич.
- **Scrumban** — планирование как в Scrum, исполнение и срочные задачи — по Kanban-потоку.

# Решение

Выбран **Scrumban** (двухнедельное планирование + Kanban-поток выполнения).

# Обоснование

- Сохраняем **прогнозируемость** поставок (итерации, цели спринта/релиза).
- Канбан-подход позволяет **не ломать спринт** при появлении срочных задач (инциденты, критичные дефекты).
- Подходит для платформы 24/7: можно выделить «ускоренную дорожку» (expedite) для инцидентов.
- Улучшает прозрачность: WIP-лимиты, понятный статус задач, регулярные ретро.

# Недостатки

- Требует дисциплины с WIP и приоритизацией, иначе «срочное» вытеснит «важное».
- Нужно договориться о правилах: что считается expedite, кто может повышать приоритет.

# Последствия

- Планирование — раз в 2 недели (цели, крупные элементы бэклога).
- Исполнение — Kanban-доска с WIP-лимитами и отдельной дорожкой для инцидентов.
- В CI/CD закрепляются quality gates: обязательные тесты, code review, статический анализ.
