# Контекст

Серверная часть платформы для трейдинга должна:

- обрабатывать высокочастотные запросы (котировки, стакан, уведомления);
- надёжно фиксировать торговые операции (создание ордера, исполнение, расчёт позиций);
- поддерживать WebSocket/streaming для realtime;
- обеспечивать безопасность (аутентификация, авторизация, аудит) и высокую доступность;
- интегрироваться с брокерами/банками и брокером сообщений.

Требуется выбрать технологию серверной разработки, учитывая производительность, поддержку конкурентности, зрелость экосистемы и удобство деплоя.

# Рассмотренные варианты

1. **Java (Spring)** — зрелая экосистема, много инструментов, хороша для крупных систем, но тяжелее по ресурсам.
2. **Kotlin (Spring)** — современнее Java, но те же инфраструктурные требования JVM.
3. **Go** — высокая производительность, дешёвая конкурентность (goroutines), простой деплой.
4. **Node.js (TypeScript)** — быстрый старт, хорош для BFF, но сложнее гарантировать предсказуемую производительность под высокой нагрузкой.
5. **Rust** — максимальная производительность, но выше порог входа и скорость разработки ниже.

# Решение

Выбран **Go** как основной язык разработки backend-сервисов.

# Обоснование

- **Производительность и низкие задержки** — важны для API ордеров и realtime-потоков.
- **Лёгкая конкурентность** (goroutines) — удобно для обработки множества соединений (WebSocket), фоновых воркеров и ретраев.
- **Простота развёртывания** — один бинарник, минимальные зависимости, удобен для контейнеризации.
- **Зрелая экосистема** для сетевого программирования и интеграций (HTTP/gRPC, драйверы БД, клиенты брокеров сообщений).

# Последствия

- Основные сервисы пишутся на Go (например, Order Management, Portfolio, Market Data Gateway).
- Для HTTP API используется **gin/echo**, для межсервисного взаимодействия предпочтительно **gRPC**.
- Требуется единый стиль логирования/трейсинга и стандарт для ошибок (error codes).
